# AGENTS.md — Architecture Conventions & Onboarding

This document captures the architectural decisions, coding conventions, and design patterns used in **Hotel Manager**. It serves as the single source of truth for maintaining consistency across the codebase.

---

## Project Structure

```
HotelManager.sln
└── src/
    ├── HotelManager.Domain/           ← Innermost layer
    ├── HotelManager.Application/      ← Use cases & business logic
    ├── HotelManager.Infrastructure/   ← External concerns
    └── HotelManager.API/              ← HTTP entry point
tests/
    └── HotelManager.Application.Tests/
frontend/                              ← React SPA
```

### Dependency Rules (Strict)

```
Domain  ←  Application  ←  Infrastructure  ←  API
```

- **Domain**: Zero dependencies. Contains only entities, enums, and interfaces.
- **Application**: Depends only on `Domain`. Contains services, DTOs, validators, and common utilities.
- **Infrastructure**: Depends on `Application` and `Domain`. Implements interfaces (EF Core, JWT).
- **API**: Depends on `Infrastructure`. Configures DI, middleware, controllers.
- **Tests**: Depends on `Application`, `Domain`, and `Infrastructure` (for EF Core InMemory).
- **Frontend**: Standalone SPA communicating via HTTP. No backend project references.

> A controller must never reference `IApplicationDbContext` or `ApplicationDbContext` directly.
> A service must never return `IQueryable<T>` — always materialize to `List<T>` or `Task<T>`.
> Business logic in services must not throw HTTP-specific exceptions. Use `ArgumentException`, `KeyNotFoundException`, `UnauthorizedAccessException`.

## Controller Patterns

Every controller follows this template:

```csharp
[ApiController]
[Route("api/{entities}")]
[Authorize]  // or [Authorize(Roles = "Owner")] for admin-only
public class {Entity}Controller : ControllerBase
{
    private readonly I{Entity}Service _{entity}Service;

    public {Entity}Controller(I{Entity}Service {entity}Service)
    {
        _{entity}Service = {entity}Service;
    }

    [HttpGet]  // List with optional filtering/pagination
    public async Task<IActionResult> GetAll([FromQuery] {Filter}Request filter)
    {
        var result = await _{entity}Service.GetFilteredAsync(filter);
        return Ok(result);
    }

    [HttpGet("{id}")]  // Single entity by ID
    public async Task<IActionResult> GetById(int id)
    {
        var item = await _{entity}Service.GetByIdAsync(id);
        return Ok(item);
    }

    [HttpPost]  // Create
    public async Task<IActionResult> Create([FromBody] CreateRequest request)
    {
        var item = await _{entity}Service.CreateAsync(request);
        return CreatedAtAction(nameof(GetById), new { id = item.Id }, item);
    }

    [HttpPatch("{id}/action")]  // State transition → 204
    public async Task<IActionResult> Action(int id)
    {
        await _{entity}Service.ActionAsync(id);
        return NoContent();
    }

    [HttpPut("{id}")]  // Full update → 200
    public async Task<IActionResult> Update(int id, [FromBody] UpdateRequest request)
    {
        var item = await _{entity}Service.UpdateAsync(id, request);
        return Ok(item);
    }

    [HttpDelete("{id}")]  // Delete → 204
    public async Task<IActionResult> Delete(int id)
    {
        await _{entity}Service.DeleteAsync(id);
        return NoContent();
    }
}
```

### Response Rules

| Service Method Result | Controller Response |
|---|---|
| Returns DTO | `Ok(dto)` → 200 |
| Creates entity | `CreatedAtAction(nameof(GetById), new { id }, dto)` → 201 |
| Void / state change | `NoContent()` → 204 |
| `KeyNotFoundException` | Middleware → 404 |
| `ArgumentException` | Middleware → 400 |
| `UnauthorizedAccessException` | Middleware → 401 |

## Service Patterns

### Stateless Services

All services are **scoped** and **stateless** — no mutable fields, no static state.

```csharp
public class BookingService : IBookingService
{
    private readonly IApplicationDbContext _context;

    public BookingService(IApplicationDbContext context)
    {
        _context = context;
    }
}
```

### Error Handling

Services throw standard .NET exceptions — never HTTP-aware types:

| Situation | Exception |
|---|---|
| Entity not found | `KeyNotFoundException` |
| Invalid input / business rule violation | `ArgumentException` |
| Authentication failure | `UnauthorizedAccessException` |

### Mapping

DTO mapping is done via **private static methods** within the service class. No AutoMapper — explicit mapping keeps things simple and traceable.

```csharp
private static BookingSummaryDto MapToSummary(Booking booking)
{
    return new BookingSummaryDto { ... };
}
```

## Validation Patterns

### FluentValidation

Every POST/PUT/PATCH request has a corresponding validator:

```
CreateGuestRequest          → CreateGuestRequestValidator
UpdateGuestRequest          → UpdateGuestRequestValidator
CreateBookingRequest        → CreateBookingRequestValidator
CreateRoomRequest           → CreateRoomRequestValidator
UpdateRoomRequest           → UpdateRoomRequestValidator
AddPaymentRequest           → AddPaymentRequestValidator
ExtendBookingRequest        → ExtendBookingRequestValidator
CreateEmployeeRequest       → CreateEmployeeRequestValidator
ChangePasswordRequest       → ChangePasswordRequestValidator
```

### Validator Rules

```csharp
public class CreateGuestRequestValidator : AbstractValidator<CreateGuestRequest>
{
    public CreateGuestRequestValidator()
    {
        RuleFor(x => x.FullName).NotEmpty().MaximumLength(100);
        RuleFor(x => x.NationalId).NotEmpty().MaximumLength(20);
        RuleFor(x => x.Address).NotEmpty().MaximumLength(300);
        RuleFor(x => x.Phone).MaximumLength(20);
    }
}
```

All validators are auto-registered and auto-applied — no manual `ModelState.IsValid` checks in controllers.

## Database Patterns

### Configuration

Entity configurations use **IEntityTypeConfiguration<T>** (Fluent API), not data annotations.

```csharp
public class BookingConfiguration : IEntityTypeConfiguration<Booking>
{
    public void Configure(EntityTypeBuilder<Booking> builder)
    {
        builder.ToTable("Bookings");
        builder.HasKey(b => b.Id);
        builder.Property(b => b.PricePerNight).HasColumnType("decimal(10,2)");
        builder.HasOne(b => b.Room)
            .WithMany(r => r.Bookings)
            .HasForeignKey(b => b.RoomId)
            .OnDelete(DeleteBehavior.Restrict);
    }
}
```

### Rules

- All foreign keys: `DeleteBehavior.Restrict` (prevents accidental cascade deletes)
- Enums stored as integers (`HasConversion<int>()`)
- Unique indexes enforced at database level where business rules require it
- Decimal properties explicitly typed: `HasColumnType("decimal(10,2)")`
- String properties have explicit `MaxLength` constraints

## Testing Patterns

### Test Structure

```
tests/HotelManager.Application.Tests/
├── Common/        ← Tests for static utility classes
├── Services/      ← Tests for application services
├── Validators/    ← Tests for FluentValidation validators
└── TestCommon/    ← Shared test utilities (MockDbContext)
```

### MockDbContext

Use `MockDbContext.CreateWithData(...)` for all service tests:

```csharp
var ctx = MockDbContext.CreateWithData(
    rooms: rooms,
    guests: guests,
    bookings: bookings,
    payments: payments,
    bookingGuests: bookingGuests,
    users: users
);
```

### Service Test Template

```csharp
[Fact]
public async Task {Method}_Scenario_ExpectedBehavior()
{
    // Arrange
    var ctx = MockDbContext.CreateWithData(...);
    var service = new {Entity}Service(ctx);

    // Act
    var result = await service.{Method}(...);

    // Assert
    result.Should().NotBeNull();
    result.SomeProperty.Should().Be(expected);
}
```

### Exception Test Template

```csharp
[Fact]
public async Task Method_InvalidState_Throws()
{
    var ctx = MockDbContext.CreateWithData(...);
    var service = new {Entity}Service(ctx);

    var act = () => service.{Method}(...);

    await act.Should().ThrowAsync<ArgumentException>()
        .WithMessage("Expected error message.");
}
```

### Naming Rules

- Test method: `{MethodName}_{Scenario}_{ExpectedBehavior}`
- Test class: `{Entity}ServiceTests` / `{Request}ValidatorTests`
- Arrange-Act-Assert pattern with blank line separators
- Use FluentAssertions for all assertions

## Exception Handling Pipeline

```
Controller
  → Service (throws ArgumentException / KeyNotFoundException / etc.)
    → GlobalExceptionHandler middleware
      → RFC 7807 ProblemDetails JSON response
```

The middleware in `Middleware/GlobalExceptionHandler.cs` catches all unhandled exceptions and maps them to consistent `ProblemDetails` responses. Unhandled `Exception` types return 500 with no details leaked to the client.

## Authentication Flow

```
Login Request
  → AuthService.LoginAsync
    → Validate credentials (BCrypt.Verify)
      → Generate JWT (JwtTokenService.GenerateToken)
        → Return { token, role, expiresAt }

Subsequent Requests
  → Axios interceptor attaches Bearer token
    → JWT middleware validates (issuer, audience, signature, expiry)
      → ClaimsPrincipal populated
        → [Authorize] / [Authorize(Roles = "...")] enforced
```

## How to Add a New Feature

### Backend (New Entity)

1. **Domain**: Create entity class in `HotelManager.Domain/Entities/`
2. **Domain**: Add enums in `HotelManager.Domain/Enums/` if needed
3. **Infrastructure**: Create entity configuration in `HotelManager.Infrastructure/Persistence/Configurations/`
4. **Infrastructure**: Add `DbSet<T>` to `ApplicationDbContext`
5. **Infrastructure**: Create EF Core migration (`dotnet ef migrations add ...`)
6. **Application**: Create DTOs (request + response)
7. **Application**: Create service interface in `Services/Interfaces/`
8. **Application**: Create service implementation in `Services/`
9. **Application**: Create validator if accepting POST/PUT requests
10. **API**: Register service in `ServicesExtensions.cs`
11. **API**: Create controller
12. **Tests**: Add service and validator tests

### Frontend (New Page)

1. Create page component in `frontend/src/pages/`
2. Add route in `App.jsx`
3. Add API calls via the shared `client` instance (`frontend/src/api/client.js`)

## Configuration

### Configuration Layering

```
appsettings.json                  ← Shared base config (no secrets)
appsettings.Development.json      ← Local dev overrides (gitignored)
Environment variables             ← Production secrets (ConnectionStrings__DefaultConnection, Jwt__Key)
dotnet user-secrets               ← Dev alternative (recommended)
```

`appsettings.json` contains only non-sensitive defaults. Secrets are never checked into source control.

### appsettings.json (shared — committed to git)

```json
{
  "Jwt": {
    "Issuer": "HotelManager.API",
    "Audience": "HotelManager.Client",
    "ExpiryHours": "8"
  },
  "Serilog": { ... },
  "FrontendUrl": "http://localhost:5174"
}
```

### appsettings.Development.json (local — gitignored)

Location: `src/HotelManager.API/appsettings.Development.json`

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=.\\SQLEXPRESS;Database=HotelManager;Trusted_Connection=True;TrustServerCertificate=True"
  },
  "Jwt": {
    "Key": "Your-32-char-dev-secret-key-here!!"
  }
}
```

### Production: Environment Variables

| Variable | Required | Description |
|---|---|---|
| `ConnectionStrings__DefaultConnection` | ✅ Yes | SQL Server connection string |
| `Jwt__Key` | ✅ Yes | HMAC-SHA256 signing key (min 32 chars) |
| `Jwt__Issuer` | ❌ No (optional override) | Override `appsettings.json` |
| `Jwt__Audience` | ❌ No (optional override) | Override `appsettings.json` |
| `Jwt__ExpiryHours` | ❌ No (optional override) | Override `appsettings.json` |
| `FrontendUrl` | ❌ No (optional override) | Override `appsettings.json` |
| `DefaultAdminPassword` | ❌ No (default: `Admin123!`) | First-run admin seed password |

> **Important**: ASP.NET Core uses `__` (double underscore) as the key separator in environment variables (e.g., `ConnectionStrings__DefaultConnection`). This works on all platforms.

### Development Alternative: User Secrets

```bash
dotnet user-secrets init --project src/HotelManager.API
dotnet user-secrets set "ConnectionStrings:DefaultConnection" "Server=.\\SQLEXPRESS;Database=HotelManager;Trusted_Connection=True;TrustServerCertificate=True"
dotnet user-secrets set "Jwt:Key" "Your-32-char-dev-secret-key-here!!"
```

User Secrets store values in a JSON file under your user profile (`%APPDATA%\Microsoft\UserSecrets\`) — never committed, scoped per developer.
